## 回答草案

### A-Q-20250919-001
- **回答**: 本仕様の解釈は **「同時保有上限=1冊」** とする。操作単位としても **1リクエスト=1冊** に限定する（UI/APIともに複数冊同時貸出は不可）。
- **根拠**: `spec_v1-1.md` に同時保有上限/操作単位の明記が見当たらないため（定義欠落）。最小介入で衝突を避けるため、在庫計算とバリデーションを単純化する方針を暫定採用。
- **リスク**: 家族代理貸出/まとめ借り等の業務要求が後出しで来た場合に差分が発生（将来の上限拡張・バッチ貸出要件に備えた拡張ポイント設計が必要）。

### A-Q-20250919-002
- **回答**: ユーザーが既に1冊保有している場合は **新規貸出を拒否** する。  
  - バリデーション: `currentLoansCount(userId) &gt;= 1` の場合はエラー。  
  - 返却後は即時に貸出可能。  
  - API 失敗コード（例）: `BORROW_LIMIT_EXCEEDED`（HTTP 409）。
- **根拠**: Q-20250919-001 の決定に従属。矛盾のない運用のため同時保有上限を厳密に適用。
- **リスク**: 管理者特例や緊急時の手当て（上限無視）が必要な場合、例外ルールの別途設計が必要。

### A-Q-20250919-003
- **回答**: 貸出期間14日を超過しても **自動返却はしない**（未実装）。貸出レコードを `status=OVERDUE` に設定し、**新規貸出は不可** とする。返却操作は常に可能。延滞金や自動停止等は **仕様範囲外**（未定義のまま維持）。
- **根拠**: `spec_v1-1.md` に期限超過時の自動処理の明記が見当たらないため。最小介入（自動処理を導入しない）で整合性のみ確保。
- **リスク**: 放置貸出が滞留する可能性。運用通知やレポート等での補完が必要（将来の拡張候補として別途検討）。
